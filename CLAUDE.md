# Key-Store 项目开发规范

## 1. 操作范围限制

- **仅允许操作本项目目录（`E:\changjiang-project\key-store`）内的文件。**
- 禁止访问、读取或修改该目录以外的任何文件或路径。
- 如需操作外部资源，必须先向用户说明原因并获得明确许可。

## 2. 远程仓库提交限制

- **在没有收到用户明确提交指令之前，禁止向远程仓库提交任何文件。**
- 禁止执行 `git push`、`git push --force` 等任何推送操作。
- `git add`、`git commit` 等操作同样需要用户明确授权后方可执行。

## 3. 执行前人工审查机制

- **每次执行前，须将本次执行内容或执行方案提交给用户进行人工审查。**
- 等待用户确认后方可开始执行，严禁未经确认直接操作文件。
- 审查内容包括但不限于：变更范围、涉及文件、预期效果、潜在风险。

## 4. 代码注释规范

编写或修改的代码必须包含清晰、完整的注释，具体要求如下：

### 4.1 类/接口注释

- 每个类、接口、枚举类必须在文件头部使用 Javadoc 风格注释说明其用途。
- 标注作者、创建时间、版本号（如有）。

### 4.2 方法/函数注释

每个方法/函数必须包含以下信息：

```java
/**
 * 方法功能简述
 *
 * @param paramName 参数说明
 * @return 返回值说明
 * @throws SpecificException 异常说明及触发条件
 */
public ReturnType methodName(ParamType paramName) throws SpecificException {
    // 方法体
}
```

必须明确说明：

| 要素 | 说明 |
|------|------|
| **函数作用** | 该方法/函数的功能简述 |
| **入参** | 每个参数的名称、类型、含义、约束条件 |
| **出参/返回值** | 返回值的类型、含义、可能的值 |
| **调用方** | 该方法被哪些外部类/方法调用（或标注调用关系） |
| **异常处理** | 可能抛出的异常类型及触发条件 |

### 4.3 关键逻辑注释

- 复杂业务逻辑、算法步骤、条件分支处需添加行内注释说明。
- 魔法数字（Magic Number）必须提取为常量并附注释说明其含义。
- 临时注释（如 `TODO`、`FIXME`、`HACK`）必须标注原因和计划修改时间。

### 4.4 常量/配置注释

- 枚举值、配置项、静态常量必须注释说明其用途和取值范围。
